```
당신은 시니어 프로덕트 매니저이자 풀스택 개발자입니다.

## 역할 및 목표

노션(Notion)에서 입력한 견적서 데이터를 클라이언트가
웹으로 확인하고 PDF로 다운로드할 수 있는 서비스의
MVP PRD(Product Requirements Document) 문서를 작성하세요.

## 프로젝트 컨텍스트

### 서비스 정보
- 서비스명: Invoce (인보이스 웹)
- 목적: 프리랜서 / 소규모 에이전시가 노션으로 작성한 견적서를
        클라이언트에게 웹 링크로 전달 → 열람 및 PDF 저장

### 기술 스택 (반드시 준수)
| 구분 | 기술 |
|------|------|
| 프론트엔드 프레임워크 | Next.js 16 (App Router) — 주의: 학습 데이터와 다른 breaking changes 존재 |
| 런타임 | React 19 |
| 스타일링 | Tailwind CSS |
| UI 컴포넌트 | shadcn/ui |
| 백엔드 프레임워크 | Spring Boot (Java) |
| 데이터베이스 | PostgreSQL |
| 외부 연동 | Notion API |
| 아키텍처 패턴 | Layered Architecture (Controller → Service → Repository) |

### 프로젝트 디렉토리 구조 (현재 상태)
```
app/
  (auth)/          # 인증 라우트 그룹
  (dashboard)/     # 관리자 대시보드 라우트 그룹
  (marketing)/     # 랜딩 페이지 라우트 그룹
  layout.tsx
  page.tsx
```

### 코딩 규칙
- 응답 언어: 한국어
- 들여쓰기: 2칸
- 네이밍: camelCase
- DTO 패턴 사용
- 의존성 주입 (Spring @Bean, @Service, @Repository 등)
- API 응답 형식 일관성 유지
- DB 트랜잭션 처리 필수

### 사용자 유형
- **관리자(Admin)**: 노션에서 견적서 작성 및 관리, 링크 발송
- **클라이언트(Client)**: 링크로 견적서 열람 및 PDF 다운로드 (회원가입 불필요)

## PRD 문서 작성 지침

아래 섹션을 순서대로 빠짐없이 작성하세요.

---

### 섹션 1. 문서 메타 정보

문서 최상단에 아래 표를 포함하세요:

| 항목 | 내용 |
|------|------|
| 문서 버전 | v0.1 |
| 작성일 | 오늘 날짜 |
| 상태 | Draft |
| 작성자 | AI PM |
| 검토자 | — |

---

### 섹션 2. 개요 (Overview)

- 서비스 한 줄 설명
- 해결하는 문제 (Pain Point)
- 핵심 가치 제안 (Value Proposition)
- 경쟁 서비스와의 차별점

---

### 섹션 3. 사용자 및 이해관계자 (Users & Stakeholders)

각 사용자 유형에 대해 다음을 작성하세요:
- 페르소나 이름과 설명
- 핵심 목표 (Jobs to be Done)
- 주요 불편함 (Frustrations)

---

### 섹션 4. MVP 범위 정의 (Scope)

두 개의 표로 나눠 작성:
- **In Scope**: MVP에 포함할 기능
- **Out of Scope**: MVP 이후로 미루는 기능 (이유 포함)

---

### 섹션 5. 기능 요구사항 (Functional Requirements)

각 기능을 아래 형식으로 작성하세요:

**FR-{번호}: {기능명}**
- 설명
- 사용자 스토리: "As a {사용자}, I want to {행동}, so that {목적}"
- 인수 조건 (Acceptance Criteria): 번호 목록으로 3개 이상

반드시 포함할 기능 목록:
1. 노션 데이터베이스 연동 (Notion API → PostgreSQL 동기화)
2. 견적서 고유 링크 생성 및 공유 (slug 또는 UUID 기반)
3. 웹 견적서 뷰어 (반응형, 모바일 대응)
4. PDF 다운로드 (서버사이드 렌더링 기반)
5. 견적서 상태 관리 (초안 / 발송 / 확인 / 승인 / 거절)
6. 관리자 대시보드 (견적서 목록, 상태 필터)
7. 클라이언트 열람 추적 (최초 열람 시각 기록)

---

### 섹션 6. 비기능 요구사항 (Non-Functional Requirements)

아래 항목별로 구체적 수치나 기준을 포함해 작성:
- 성능 (응답 시간, PDF 생성 시간)
- 보안 (링크 만료, 접근 제어)
- 접근성 (WCAG 기준)
- 확장성
- 유지보수성

노션 프로퍼티 이름과 PostgreSQL 컬럼명을 나란히 표시하세요.

### 섹션 8. API 설계 요약

아래 표 형식으로 작성:

| 메서드 | 경로 | 설명 | 인증 필요 |

### 섹션 9. UI/UX 플로우

#### 9-1. 관리자 플로우
노션 작성 → 동기화 → 링크 발송 → 상태 확인까지 단계별로 기술

#### 9-2. 클라이언트 플로우
링크 수신 → 웹 열람 → PDF 다운로드 → 승인/거절까지 단계별로 기술

각 플로우는 번호가 있는 단계로 작성하고,
각 단계에서 사용되는 페이지/컴포넌트 이름을 괄호로 병기하세요.
```
